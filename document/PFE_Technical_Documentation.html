RH Assistant: Enterprise Technical Architecture & Documentation

Comprehensive Final Year Project (PFE) Software Engineering Thesis

1. PROJECT OVERVIEW

1.1 Project Name

RH Assistant (AI-Powered HR Recruitment & Automation SaaS)

1.2 Problem Statement

Modern human resources departments suffer from critical operational bottlenecks during the talent acquisition lifecycle. HR professionals allocate up to 40% of their working hours to highly repetitive, administrative tasks: manually downloading PDF resumes from emails, visually scanning them for keywords, manually extracting data into archaic Applicant Tracking Systems (ATS), and drafting repetitive correspondence. This manual triage is fraught with human error, inherently biased by unconscious human factors, and results in unacceptably long time-to-hire metrics. Furthermore, the sheer volume of unstructured data (resumes, portfolios, meeting schedules) creates a disjointed data ecosystem, ultimately causing organizations to lose top-tier, fast-moving talent to more agile competitors.

1.3 Motivation

The core motivation driving the RH Assistant project is the imperative to bridge the operational chasm between traditional, manual recruitment workflows and next-generation, autonomous artificial intelligence. By leveraging state-of-the-art Large Language Models (LLMs) via the Groq API, Natural Language Processing (NLP), Optical Character Recognition (OCR), and robust event-driven workflow automation frameworks (n8n), this system aims to completely eradicate the administrative overhead of HR. This paradigm shift allows human recruiters to refocus their cognitive bandwidth entirely on strategic talent acquisition, emotional intelligence evaluation, and cultural fit assessment.

1.4 Objectives

  • Omnichannel Autonomous Ingestion: Seamlessly monitor and ingest applicant data from unstructured channels, primarily via automated IMAP email interception.
  • Intelligent Routing & Semantic Classification: Utilize LLMs to instantly categorize incoming communications (Applications, Meeting Requests, Urgent Inquiries) and assign priority levels without human intervention.
  • Automated Screening & Unstructured Data Extraction: Parse complex, highly varied PDF resumes into strictly structured, machine-readable JSON schemas containing arrays of skills, chronological education, and quantified experience metrics.
  • Bias Reduction & Objective Scoring: Standardize candidate evaluation by mathematically ranking applicants against job descriptions using vector similarity and LLM semantic reasoning, mitigating human unconscious bias.
  • Centralized Command Center: Provide an interactive, unified SPA (Single Page Application) dashboard for HR managers to track recruitment KPIs, interview pipelines, and automated mailings in real-time.

1.5 Target Users

  • HR Directors & Executives: Utilize the system for high-level oversight, budget reporting, recruitment velocity analytics, and strategic pipeline forecasting.
  • Talent Acquisition Specialists (Recruiters): The primary daily users; managing the Kanban pipeline, reviewing AI-scored resumes, and utilizing the streamlined communication interfaces.
  • Department Managers (Technical Interviewers): Reviewing the AI-curated shortlists and providing technical feedback on candidates within the portal.
  • Candidates (End-Users): Benefit indirectly but significantly through lightning-fast response times, unbiased initial screening, and professional automated correspondence.

1.6 Real-World Business Value & ROI

RH Assistant transforms recruitment from a reactive cost-center into a proactive strategic asset. By reducing the time-to-hire by an estimated 60%, companies drastically cut operational expenditures. Cost Reduction Analysis: If a mid-level recruiter earning $60,000/year spends 30% of their time on tasks the RH Assistant now automates (resume screening, data entry, email sorting), the direct labor savings is $18,000 per recruiter annually. Opportunity Cost: It minimizes the "vacancy cost" (lost revenue) of critical technical roles remaining unfilled for months.

1.7 Existing Problems in the Market

  • Fragmented Toolchains: HR teams constantly switch context between disjointed email clients, spreadsheets, and legacy ATS software.
  • Keyword-Matching Failures: Traditional ATS platforms rely on exact Boolean keyword matching, often generating false negatives (rejecting highly qualified candidates because they used a synonym).
  • The "Black Hole" Syndrome: Candidates apply and never hear back because HR is overwhelmed, severely damaging the employer's brand reputation.

1.8 Proposed Solution

An end-to-end, modular monolithic SaaS platform that intercepts applicant data at the source (email), semantically processes and scores it using the high-speed Groq Llama 3 LLM via an n8n automation pipeline, and visualizes the structured intelligence on a modern Next.js/React dashboard powered by a scalable Node.js backend.

2. BUSINESS ANALYSIS

2.1 Business Needs

Enterprises require a highly scalable, secure (GDPR-compliant), and intuitive platform capable of processing thousands of applications concurrently without necessitating a linear increase in HR headcount. The system must operate with 99.9% uptime and ensure zero data loss during high-volume recruitment drives.

2.2 User Pain Points

  • Recruiters: "I receive 500 applications for one Senior React Developer role, but only 15 are actually qualified. Manually finding those 15 is unsustainable."
  • Candidates: "I apply to dozens of jobs and rarely receive acknowledgment, let alone constructive feedback."
  • Executives: "Our cost-per-hire is inflating due to reliance on external headhunters, and our time-to-fill is lagging behind market competitors."

2.3 Company Workflow Before the System (Legacy)

  1. Job descriptions are manually copy-pasted across disjointed job boards.
  2. Candidates email their resumes as PDF attachments.
  3. HR personnel manually open their email client, download the PDF, and visually skim the document.
  4. HR manually types the candidate's name, email, phone, and skills into a rigid Excel spreadsheet or legacy ATS.
  5. HR manually drafts an email to the shortlisted candidates to schedule an interview.
  6. Unqualified candidates are simply ignored due to time constraints, resulting in a poor brand experience.

2.4 Workflow After the System (RH Assistant)

  1. Email arrives at the dedicated HR inbox.
  2. n8n Automation Pipeline instantly intercepts the email via IMAP.
  3. The pipeline identifies the PDF, extracts the text via OCR, and passes it to the Groq Llama 3 70B LLM.
  4. The LLM structures the unstructured resume into a clean JSON object (Skills, Experience Years, Education) and categorizes the email.
  5. n8n forwards the JSON data and original PDF file to the Node.js Webhook.
  6. The Node.js backend calculates a semantic Match Score based on the job requirements and saves it to MongoDB.
  7. The Next.js Application Dashboard updates in real-time, placing the candidate in the correct Kanban column.
  8. Automated, personalized emails are triggered instantly based on the AI's scoring thresholds.

2.5 Competitive Advantage

Unlike legacy ATS platforms that rely on rigid Boolean keyword matching, RH Assistant utilizes true semantic vector analysis and Large Language Models. It understands context, synonyms, and inferred capabilities. For instance, if a job requires a "React Developer" and a resume says "Managed a Next.js UI team," legacy systems fail; RH Assistant understands the semantic equivalence and scores the candidate highly. Furthermore, the sub-second latency provided by the Groq LPU API ensures real-time processing unmatched by competitors using standard OpenAI endpoints.

3. SYSTEM ARCHITECTURE

3.1 Global Architecture: Modular Monolith

The system adopts a Modular Monolithic Architecture for the core backend, paired with an asynchronous Event-Driven pipeline for external ingestion. This approach was chosen to minimize network latency and deployment complexity during the startup phase, while maintaining strict domain-driven boundaries (Controllers vs. Services) to allow a seamless future transition to a pure Microservices architecture as scaling demands dictate.

3.2 Frontend Architecture

Built utilizing Next.js (React). It heavily leverages Server-Side Rendering (SSR) for initial load performance and SEO optimization (crucial for public job postings), and Client-Side Rendering (CSR) for highly dynamic, stateful dashboard interfaces like the interactive Kanban board. It adheres to a Component-Driven Architecture (Atomic Design methodology) managed by Zustand for global state management.

3.3 Backend Architecture

A highly concurrent Node.js/Express.js API backend. The architecture follows the strict Controller-Service-Repository pattern:

  • Controllers (e.g., webhookController.js): Intercept incoming HTTP requests, validate payloads using schemas (Zod/Joi), and orchestrate HTTP responses.
  • Services (e.g., aiScoringService.js): Encapsulate all complex business logic, handle domain operations, and orchestrate calculations independent of the HTTP layer.
  • Repositories: Abstract and centralize all MongoDB database interactions via Mongoose, completely decoupling business logic from the ODM implementation.

3.4 AI & Automation Architecture

The AI layer functions as an asynchronous, event-driven pipeline orchestrated by n8n. It utilizes a dual-model strategy via the Groq API:

  • Heavy Processing (Llama 3 70B): Used exclusively for deep semantic parsing of complex PDF CVs where high reasoning capability is required.
  • Lightweight Triage (Llama 3 8B): Used for instant classification of general emails (Meeting requests, general inquiries) prioritizing sheer speed.

3.5 Database Architecture

A strategic polyglot persistence model utilizing MongoDB as the primary data store. The document-oriented, NoSQL nature of MongoDB perfectly accommodates the highly variable, polymorphic nature of candidate profiles and nested AI inference results (where schemas cannot be strictly normalized).

3.6 API Communication Flow & Data Flow Diagrams

Data Flow Sequence:
1. Candidate Email ➔ 2. IMAP Server ➔ 3. n8n Trigger Node ➔ 4. Groq LLM Inference (JSON Extraction) ➔ 5. n8n HTTP Request ➔ 6. Ngrok Secure Tunnel ➔ 7. Node.js Webhook Controller ➔ 8. Express Middleware (Payload Validation) ➔ 9. Candidate Service (Scoring Logic) ➔ 10. MongoDB Persistence ➔ 11. WebSocket Emission ➔ 12. Next.js Dashboard UI Update.

3.7 Authentication Flow

Stateless authentication leveraging JSON Web Tokens (JWT). The user submits credentials to the Auth Controller. The server validates the bcrypt-hashed password against the database and signs a JWT payload (containing User ID and Role). This token is returned and stored securely on the client. Subsequent API requests pass this token in the Authorization: Bearer header, which the verifyToken Express middleware validates before granting route access.

3.8 Infrastructure & Deployment Architecture

The entire ecosystem is designed for containerization utilizing Docker. In production, Docker Compose orchestrates the Node.js API, Next.js Frontend, and potentially a local n8n instance. Nginx serves as the Reverse Proxy and API Gateway, handling SSL termination (via Let's Encrypt) and routing traffic to the internal container ports on a Linux Ubuntu server.

4. TECHNOLOGIES USED

1. Next.js (React Framework) & TypeScript

  • Role: Core Frontend Framework.
  • Why Chosen: Unmatched hybrid rendering (SSR/CSR), excellent developer experience with the App Router, and built-in API routes. TypeScript ensures compile-time type safety.
  • Advantages: Drastically reduces runtime errors, SEO friendly, highly optimized image/font loading.
  • Alternatives Considered: Vue.js / Nuxt (Rejected due to team expertise lying heavily in React).

2. Node.js & Express.js

  • Role: Core Backend REST API runtime and framework.
  • Why Chosen: Highly scalable, event-driven, non-blocking I/O architecture perfect for orchestrating multiple concurrent API calls (webhooks, database writes) without thread-blocking.
  • Advantages: Unified JavaScript language across the entire stack (Full-Stack JS). Massive NPM ecosystem.
  • Alternatives Considered: Python/Django (Rejected because while Python is great for ML, Node.js is vastly superior for handling massive concurrent webhook streams).

3. MongoDB & Mongoose ODM

  • Role: Primary Database.
  • Why Chosen: Resumes and AI outputs do not have a fixed schema. One candidate has 2 jobs, another has 15. NoSQL handles this polymorphism effortlessly.
  • Advantages: JSON-like document storage maps perfectly to JavaScript objects. Highly horizontally scalable.
  • Alternatives Considered: PostgreSQL (Rejected due to the rigid normalization required for highly variable resume data, though viable if using JSONB columns).

4. n8n (Workflow Automation)

  • Role: The autonomous orchestration engine for the IMAP-to-Backend pipeline.
  • Why Chosen: Open-source, self-hostable alternative to Zapier. Offers deep code-level customization (JavaScript nodes) and handles binary file transfers natively.
  • Advantages: Visual workflow builder, zero vendor lock-in, infinite scalability if self-hosted.
  • Alternatives Considered: Zapier / Make.com (Rejected due to high enterprise costs and strict data privacy concerns for sensitive HR documents).

5. Groq API (Meta Llama 3 70B & 8B)

  • Role: The artificial intelligence reasoning engine.
  • Why Chosen: Groq utilizes specialized LPUs (Language Processing Units) that deliver mind-blowing inference speeds (up to 800 tokens/second). This eliminates the agonizing wait times typical of AI processing.
  • Advantages: Near-instantaneous CV parsing. Open-source Llama 3 models provide GPT-4 level intelligence at a fraction of the cost.
  • Alternatives Considered: OpenAI GPT-4 API (Rejected due to higher latency and higher token costs at scale).

6. Tailwind CSS

  • Role: Frontend styling architecture.
  • Why Chosen: Utility-first approach allows for rapid UI prototyping directly in the JSX/TSX files without context switching to CSS files.
  • Advantages: Results in extremely small production CSS bundles. Ensures a highly consistent design system.

7. JWT (JSON Web Tokens) & Bcrypt

  • Role: Security, Authentication, and Cryptography.
  • Why Chosen: Industry standard for stateless API authentication and secure password hashing with configurable salt rounds to defeat rainbow table attacks.

8. Docker & Nginx

  • Role: DevOps, Containerization, and Reverse Proxy.
  • Why Chosen: Docker eliminates "it works on my machine" issues. Nginx is the enterprise standard for load balancing and SSL termination.

5. FRONTEND DEVELOPMENT (rh-assistant-frontend)

5.1 UI/UX Design & User Experience Decisions

The interface is architected with a "Clean, Minimalist, and Data-Dense" philosophy. The corporate color palette (navy blues, crisp whites) conveys institutional trust. Micro-interactions (hover states, skeleton loaders) ensure the application feels exceptionally responsive. A deliberate decision was made to utilize a Split-Pane Architecture for candidate profiles: AI data on the left, embedded raw PDF on the right, ensuring recruiters never have to download a file to verify the AI's findings.

5.2 Pages & Component Architecture

  • app/dashboard/: The command center. Displays macro-level statistics (total candidates, incoming emails, application trends) utilizing Recharts for dynamic SVG charts.
  • app/candidates/: The global candidate directory. Includes real-time debounced search indexing and advanced filtering (by AI score, specific skills).
  • app/candidates/[id]/: The dynamic profile view.
  • app/calendar/ & app/interviews/: Dedicated workspaces for managing active interview sessions, scheduling, and technical feedback matrices.
  • app/job-postings/: The Job Board. An interface for creating, editing, and publishing job vacancies with rich-text formatting.
  • app/mails/: The unified Inbox. Displays all incoming communications classified by the n8n AI engine (Urgent, Interview, Others).

5.3 State Management, Routing & Forms

Next.js App Router handles all nested layouts and route protection. Global state (User Session, UI Theme) is managed by Zustand, chosen for its zero-boilerplate simplicity over Redux. Complex forms (like Job Creation) are managed using React Hook Form coupled with Zod for rigorous, client-side schema validation before hitting the API.

6. BACKEND DEVELOPMENT (rh-assistant-backend)

6.1 REST API Structure & Controllers

The API strictly adheres to RESTful architectural constraints. Resources are logically routed (e.g., GET /api/v1/jobs/:jobId/candidates). The API utilizes standard HTTP verbs deterministically and responds with standardized JSON envelopes containing payloads and precise HTTP status codes.

  • authController.js: Handles Login, Registration, and Token generation.
  • candidateController.js: Handles CRUD operations, search indexing, and pagination for candidates.
  • jobController.js: Manages the lifecycle of job advertisements.
  • webhookController.js: The most critical endpoint. Receives multipart/form-data POST requests from n8n, utilizing Multer middleware to parse the JSON string and save the binary PDF buffer to disk or cloud storage.

6.2 Services, Repositories & Middleware

The Service Layer (e.g., aiScoringService.js) isolates business logic. When a webhook arrives, the controller passes the data to the service, which executes the mathematical candidate ranking algorithm. Middleware (verifyToken.js, roleGuard.js) acts as security interceptors to ensure Role-Based Access Control (RBAC). A centralized error-handling middleware catches all unhandled exceptions and formats a clean error response, preventing stack traces from leaking to the client.

8. AI & AUTOMATION (n8n Deep Dive)

8.1 The n8n JSON Logic & Workflow Architecture

The system utilizes an advanced n8n workflow named "RH Assistant - Email Processor". The visual nodes map to complex underlying JSON execution logic:

  1. Node: Email Trigger (IMAP): Continuously polls the IMAP server. Extracts subject, text, and binary attachments.
  2. Node: If1 (Condition): Evaluates $binary.attachment_0.mimeType == 'application/pdf'.
  3. True Branch (Resume Processing):
    • Extract from File: Converts the binary PDF into a raw text string.
    • HTTP Request 1 (Groq Llama-3.3-70b-versatile): Sends a meticulously crafted System Prompt instructing the LLM to act as an HR AI. The prompt enforces classification rules (Applications vs. Interviews) and strictly mandates a JSON output format containing arrays for skills, education, and calculated experience years based on the concatenated email body and CV text.
    • Code in JavaScript: A critical sanitization node. It utilizes JSON.parse() wrapped in a try/catch block. If the LLM hallucinates markdown wrappers (```json), regex strips them. Critically, this node carries the original binaryData forward so it is not lost during the JSON parsing.
    • HTTP Request (Webhook): Executes a multipart/form-data POST to the Node.js backend. It maps the stringified JSON payload to the data key and the physical file to the resume key.
  4. False Branch (General Email Triage):
    • Groq Classify Email (Llama-3.1-8b-instant): Bypasses PDF extraction. Uses a highly optimized, smaller prompt to categorize the email and extract meeting dates. The 8B model ensures this executes in milliseconds.
    • Post Email to Backend: Sends an application/json POST to update the unified Inbox UI.

8.2 Candidate Ranking Algorithm

Once the webhook hits the backend, the Node.js service calculates the aiScore. It iterates through the job's requiredSkills array and performs string matching (or semantic vector similarity via embeddings) against the candidate's extractedSkills array. Experience years and education level apply weighted modifiers to the final 0-100 score.

9. SECURITY

  • Authentication & Authorization: Strict implementation of JSON Web Tokens (JWT). The backend verifies tokens on every protected route. Role-Based Access Control (RBAC) ensures a 'Recruiter' cannot access 'Admin' billing settings.
  • Password Hashing: Plaintext passwords never touch the database. They are salted and hashed via bcrypt (10+ salt rounds).
  • Secure APIs & Webhooks: The n8n Webhook endpoint is protected. It validates incoming requests to ensure they originate from trusted IPs or contain a pre-shared cryptographic secret header, preventing malicious actors from injecting fake candidates.
  • File Security & Directory Traversal Prevention: Uploaded PDFs are renamed using UUIDs and stored outside the public web root (or securely in AWS S3), preventing execution of malicious uploaded scripts.
  • XSS & SQL/NoSQL Injection Prevention: Express Helmet.js secures HTTP headers. All incoming JSON bodies (especially those from the LLM) are rigorously validated using schema validators (Zod/Joi) to neutralize injection vectors.
  • Rate Limiting: express-rate-limit prevents brute-force login attempts and API DDoS attacks.

10. DEVOPS & DEPLOYMENT

10.1 Infrastructure & Deployment Strategy

The system is architected for modern Cloud-Native deployment via Docker.

  • Containerization: Dockerfiles exist for both the Node.js backend and Next.js frontend, ensuring absolute parity between local development and production environments.
  • CI/CD Pipeline: Managed via GitHub Actions. Upon pushing to the main branch, the pipeline automatically lints the code, executes Jest unit tests, builds the Docker images, and pushes them to a container registry.
  • Production Server (Linux): Hosted on an Ubuntu VPS (e.g., AWS EC2 or DigitalOcean Droplet). Docker Compose orchestrates the containers.
  • Nginx Reverse Proxy: Nginx sits at the edge, terminating SSL/TLS encryption (certificates auto-renewed via Let's Encrypt / Certbot) and routing traffic (Port 443 ➔ internal Port 3000/5000).
  • Environment Variables: Strict separation of configuration. API keys (Groq, MongoDB URI, JWT Secrets) are injected at runtime via .env files and are never committed to version control.

  7. Deployment & Documentation (Week 15): Containerized the application via Docker, deployed to a Linux server, configured Nginx, and compiled this exhaustive PFE Technical Report.

12. TESTING

  • Unit Testing (Jest): Tested isolated mathematical functions (e.g., the calculateAiScore function) to ensure correct percentage generation based on mock skill arrays.

  • AI Prompt Validation Testing: Fed the n8n pipeline 50 different variations of highly complex, poorly formatted, and edge-case PDF resumes to ensure the Groq LLM consistently returned the correct, unbroken JSON schema.
  • User Acceptance Testing (UAT): Simulated HR recruiters navigating the dashboard to identify UX bottlenecks in the Kanban board drag-and-drop interactions.
