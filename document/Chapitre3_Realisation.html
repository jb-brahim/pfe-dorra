Chapitre 3 : Réalisation & Implémentation Détaillée

Introduction au Cycle de Développement

Ce chapitre est dédié à la phase cruciale de la réalisation technique de la plateforme « RH Assistant ». Cette phase traduit les concepts théoriques, les diagrammes architecturaux et les maquettes en un produit logiciel fonctionnel, robuste et prêt pour la production. Conscient de l'envergure du projet et de la nécessité d'intégrer des technologies de pointe (Intelligence Artificielle Générative, Orchestration de flux, Développement Web Full-Stack), nous avons adopté une méthodologie Agile Scrum stricte.

La méthodologie Scrum nous a permis de décomposer la complexité du projet en itérations courtes et gérables appelées Sprints. Chaque sprint a duré exactement deux semaines. À la fin de chaque sprint, un incrément logiciel fonctionnel et testé a été livré. Cette approche itérative a été particulièrement vitale lors de l'intégration des API d'Intelligence Artificielle, où le comportement non déterministe des modèles LLM (Large Language Models) nécessitait des ajustements constants (Prompt Engineering) basés sur des tests empiriques continus.

Environnement de Développement (Stack Technique de Référence) :
  • Frontend : React 18, Next.js 14 (App Router), TypeScript, Tailwind CSS, Zustand, React-Beautiful-DnD.
  • Backend : Node.js 20, Express.js, Multer, JSON Web Tokens (JWT), Bcrypt.
  • Base de données : MongoDB Atlas, Mongoose ODM.
  • IA & Automatisation : n8n (Self-hosted), Groq API, Meta Llama-3.3-70B, Meta Llama-3.1-8B.
  • Outils de Test & DevOps : Postman, Jest, Ngrok, Docker, GitHub Actions.

Ce chapitre est structuré selon la chronologie exacte de notre développement, divisé en quatre sprints majeurs. Pour chaque sprint, nous détaillerons exhaustivement : la liste des User Stories avec leurs critères d'acceptation, la Modélisation conceptuelle et architecturale, les détails du code source d'Implémentation, et enfin les stratégies de Tests et de Validation.

La première étape a consisté à modéliser la collection Users. Nous avons défini des règles strictes au niveau du schéma Mongoose pour garantir l'intégrité des données avant même qu'elles ne touchent la base de données.

// Schéma de la collection Users { _id: ObjectId, firstName: { type: String, required: true, trim: true }, lastName: { type: String, required: true, trim: true }, email: { type: String, required: true, unique: true, lowercase: true, index: true }, passwordHash: { type: String, required: true }, role: { type: String, enum: ['ADMIN', 'HR_MANAGER', 'RECRUITER'], default: 'RECRUITER' }, isActive: { type: Boolean, default: true }, createdAt: Timestamp, updatedAt: Timestamp }

1.2.2 Diagramme de Séquence de l'Authentification JWT

Le flux d'authentification suit un modèle strict de type Challenge-Response. Le client Next.js envoie les credentials. Le backend Express.js vérifie l'existence de l'email, puis utilise l'algorithme cryptographique de bcrypt pour comparer le mot de passe en clair envoyé avec le passwordHash stocké. Si la comparaison est un succès, un payload JSON contenant l'identifiant et le rôle de l'utilisateur est signé cryptographiquement en utilisant la variable JWT_SECRET via l'algorithme HMAC SHA-256. Ce JWT est renvoyé au client, qui le stocke (idéalement dans un HttpOnly cookie ou en mémoire) et l'attache en tant que Bearer Token dans l'en-tête Authorization de toutes les requêtes HTTP futures.

1.3 Implémentation du Code Source

1.3.1 Le Contrôleur d'Authentification (Backend)

L'implémentation du contrôleur authController.js est le cœur de la sécurité du système. Voici l'extrait exact de notre code de production gérant la connexion :

// rh-assistant-backend/src/controllers/authController.js
const bcrypt = require('bcryptjs');
const jwt = require('jsonwebtoken');
const User = require('../models/User');

exports.login = async (req, res) => {
    try {
        const { email, password } = req.body;

        // 1. Validation basique des entrées
        if (!email || !password) {
            return res.status(400).json({ error: "L'email et le mot de passe sont requis." });
        }

        // 2. Recherche de l'utilisateur dans MongoDB (indexé sur l'email)
        const user = await User.findOne({ email: email.toLowerCase() });
        if (!user) {
            return res.status(404).json({ error: "Utilisateur non trouvé avec cet email." });
        }

        // 3. Comparaison cryptographique pour prévenir les attaques temporelles
        const isMatch = await bcrypt.compare(password, user.passwordHash);
        if (!isMatch) {
            return res.status(401).json({ error: "Mot de passe incorrect." });
        }

        // 4. Génération du Token JWT
        const payload = {
            userId: user._id,
            role: user.role,
            email: user.email
        };

        const token = jwt.sign(payload, process.env.JWT_SECRET, { 
            expiresIn: '24h',
            algorithm: 'HS256' 
        });
        
        // 5. Réponse sécurisée sans retourner le hash du mot de passe
        res.status(200).json({ 
            message: "Connexion réussie", 
            token: token, 
            user: { 
                id: user._id,
                firstName: user.firstName, 
                lastName: user.lastName,
                role: user.role 
            } 
        });
    } catch (error) {
        console.error("[AuthController Error]:", error);
        res.status(500).json({ error: "Erreur critique du serveur interne." });
    }
};

1.3.2 Le Middleware de Protection des Routes (Backend)

Pour protéger les autres routes de l'API (comme l'accès aux données des candidats), un middleware Express a été implémenté pour intercepter les requêtes et vérifier la validité du JWT.

// rh-assistant-backend/src/middleware/authMiddleware.js
const jwt = require('jsonwebtoken');

exports.verifyToken = (req, res, next) => {
    // Récupération de l'en-tête Authorization
    const authHeader = req.headers.authorization;
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
        return res.status(401).json({ error: "Accès refusé. Aucun token fourni." });
    }

    // Extraction du token (suppression du mot 'Bearer ')
    const token = authHeader.split(' ')[1];

    try {
        // Vérification cryptographique du token
        const verifiedPayload = jwt.verify(token, process.env.JWT_SECRET);
        // Injection du payload dans l'objet requête pour les contrôleurs suivants
        req.user = verifiedPayload; 
        next(); // Passe au middleware/contrôleur suivant
    } catch (error) {
        return res.status(403).json({ error: "Token invalide ou expiré." });
    }
};

1.3.3 L'Intercepteur Axios (Frontend Next.js)

Sur le frontend, plutôt que d'attacher manuellement le token à chaque requête (ce qui génère de la duplication de code), nous avons configuré une instance Axios globale avec des Interceptors.

// rh-assistant-frontend/lib/api.ts
import axios from 'axios';

// Création d'une instance Axios centralisée
const api = axios.create({
    baseURL: process.env.NEXT_PUBLIC_API_URL || 'http://localhost:5000/api/v1',
    headers: { 'Content-Type': 'application/json' }
});

// Intercepteur de requête : attache le token JWT automatiquement
api.interceptors.request.use((config) => {
    // Récupération du token depuis le localStorage (ou Zustand store)
    const token = typeof window !== 'undefined' ? localStorage.getItem('jwt_token') : null;
    if (token) {
        config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
}, (error) => {
    return Promise.reject(error);
});

export default api;

2.4 Implémentation du Webhook avec Multer (Backend)

La difficulté majeure de ce sprint réside dans le fait que n8n doit envoyer un fichier physique (binaire) en même temps que des données textuelles. Une requête JSON standard ne peut pas accomplir cela. L'API doit accepter le format multipart/form-data. Nous avons utilisé le middleware Multer pour intercepter le flux entrant, écrire le fichier PDF sur le disque dur, et placer les données textuelles dans req.body.

// rh-assistant-backend/src/controllers/webhookController.js
const multer = require('multer');
const path = require('path');
const Candidate = require('../models/Candidate');

// 1. Configuration approfondie du moteur de stockage Multer
const storage = multer.diskStorage({
    destination: function (req, file, cb) {
        // Le dossier où les PDF seront physiquement enregistrés
        cb(null, path.join(__dirname, '../../uploads/resumes/'));
    },
    filename: function (req, file, cb) {
        // Sécurisation du nom de fichier pour éviter les écrasements
        // Exemple: cv-16892348923-48291.pdf
        const uniqueSuffix = Date.now() + '-' + Math.round(Math.random() * 1E9);
        const ext = path.extname(file.originalname).toLowerCase();
        cb(null, 'cv-' + uniqueSuffix + ext);
    }
});

// 2. Filtre de sécurité pour n'accepter QUE les PDF
const fileFilter = (req, file, cb) => {
    if (file.mimetype === 'application/pdf') {
        cb(null, true);
    } else {
        cb(new Error("Seuls les fichiers PDF sont autorisés !"), false);
    }
};

// 3. Initialisation de Multer
const upload = multer({ 
    storage: storage,
    fileFilter: fileFilter,
    limits: { fileSize: 10 * 1024 * 1024 } // Limite stricte à 10 MB par fichier
}).single('resume'); // 'resume' est le nom du champ envoyé par n8n

// 4. Le contrôleur du Webhook
exports.handleIncomingApplication = (req, res) => {
    upload(req, res, async (err) => {
        // Gestion rigoureuse des erreurs de téléchargement
        if (err instanceof multer.MulterError) {
            return res.status(400).json({ error: "Erreur Multer: " + err.message });
        } else if (err) {
            return res.status(400).json({ error: "Erreur serveur de fichiers: " + err.message });
        }

        // Si le téléchargement a réussi, le fichier physique est sur le disque.
        try {
            // L'objet JSON envoyé par n8n est récupéré ici
            const parsedData = JSON.parse(req.body.data);
            const savedFilePath = req.file ? `/uploads/resumes/${req.file.filename}` : null;

            // Création temporaire du candidat en BDD (sera mis à jour au Sprint 3)
            const newCandidate = new Candidate({
                email: parsedData.senderEmail,
                resumeFileUrl: savedFilePath,
                source: "Email Automation"
            });
            await newCandidate.save();

            // Réponse de succès à n8n pour qu'il termine son flux sans erreur
            res.status(201).json({ message: "Webhook intercepté, fichier sauvegardé." });
        } catch (dbError) {
            console.error(dbError);
            res.status(500).json({ error: "Erreur lors de l'enregistrement en base." });
        }
    });
};

2.5 Configuration de n8n et Explication du Flux JSON

Le workflow n8n (importé via le fichier RH Assistant - Email Processor (1).json) est configuré avec les nœuds suivants :

  1. Nœud Email Trigger (IMAP) : Configuré avec l'hôte IMAP (ex: imap.gmail.com), le port 993, et SSL/TLS. L'option "Download Attachments" est activée.
  2. Nœud If : Il inspecte dynamiquement le Payload. L'expression de la condition est : {{ $binary.attachment_0 ? $binary.attachment_0.mimeType : '' }} évaluée contre la valeur application/pdf.
    • Si vrai (PDF présent), le flux passe à l'extraction de données.
    • Si faux (Email texte), le flux passe à un webhook de "Communication Générale".
  3. Nœud HTTP Request (vers le backend) : Paramétré en méthode POST. L'URL pointe vers un tunnel Ngrok (ex: https://[id-ngrok].ngrok-free.dev/api/webhooks/incoming) car n8n, fonctionnant dans le cloud ou sur Docker, ne peut pas requêter directement localhost:5000. Le corps de la requête est configuré en multipart/form-data.

3. Développement du Sprint 3 : Intelligence Artificielle (LLMs Groq) et Algorithme de Ranking

Ce sprint est la véritable innovation du projet. Il transforme une simple application de stockage de CV en un système d'intelligence sémantique. Les méthodes traditionnelles d'ATS (Applicant Tracking Systems) utilisent des expressions régulières (Regex) pour trouver des mots-clés. Nous utilisons l'IA Générative pour comprendre le CV.

Pourquoi utiliser Groq et Llama 3 ?
Contrairement aux GPU traditionnels utilisés par OpenAI (ChatGPT), l'architecture de Groq repose sur des LPU (Language Processing Units), garantissant une latence d'inférence extrêmement faible (jusqu'à 800 tokens générés par seconde). C'est crucial car n8n peut bloquer (Timeout) si l'IA met 30 secondes à analyser un CV de 5 pages. De plus, les modèles Open Source Meta Llama 3 70B (70 milliards de paramètres) rivalisent avec GPT-4 en termes de capacité de raisonnement logique.

3.1 Modélisation de l'Ingénierie de Prompt (Prompt Engineering)

L'IA doit agir comme une API déterministe. Elle ne doit pas dire "Bonjour, voici l'analyse du CV...". Elle doit retourner strictement du JSON. Voici l'instruction exacte (System Prompt) injectée dans le nœud HTTP n8n pointant vers l'API de Groq :

// System Prompt pour Llama-3.3-70b-versatile
You are a professional HR Assistant AI. Your task is to analyze the email and CV. 

CLASSIFICATION RULES:
1. If the email is a job application (CV attached), set category to 'Applications'.
2. If the email is about scheduling or confirming an interview, set category to 'Job Interview'.
3. If it is an urgent or important email from management, set category to 'Important'.
4. For everything else, use 'Others'.

PRIORITY RULES:
- High: Direct applications, interview confirmations.
- Medium: Follow-ups, general inquiries.
- Low: Newsletters, notifications.

You MUST return a valid JSON object with EXACTLY these keys: senderName, category, priority, fullName, phone, summary, extractedSkills (array of strings), education (array), experienceYears (integer), certifications, languages, portfolioLinks, recommendationLevel. DO NOT wrap the response in markdown blocks.

3.2 Implémentation du Nœud JavaScript de Nettoyage (n8n)

Malgré les instructions strictes, les LLMs peuvent parfois "halluciner" des blocs Markdown (ex: ```json { ... } ```). Si ce texte est envoyé tel quel au backend, JSON.parse() échouera lamentablement (Crash du serveur). Nous avons donc injecté un nœud Code in JavaScript dans n8n, agissant comme un pare-feu de sanitisation :

// Nœud "Code in JavaScript" dans le pipeline n8n
const aiResponse = items[0].json.choices[0].message.content;

// 1. Logique anti-hallucination Markdown
let parsed;
try {
  // Tentative de parsing direct
  parsed = JSON.parse(aiResponse);
} catch (e) {
  // En cas d'échec, nettoyage agressif par Regex
  const cleaned = aiResponse.replace(/```json\n?/g, '').replace(/```\n?/g, '').trim();
  parsed = JSON.parse(cleaned);
}

// 2. Récupération de l'objet binaire original (le fichier PDF)
const emailNode = $('Email Trigger (IMAP)').first();
const binaryData = emailNode.binary;

// 3. Reconstruction d'un objet parfait et garanti pour le Webhook
return [{
  json: {
    senderEmail: emailNode.json.from.value[0].address,
    subject: emailNode.json.subject,
    fullName: parsed.fullName,
    summary: parsed.summary,
    extractedSkills: parsed.extractedSkills || [],
    experienceYears: parsed.experienceYears || 0,
    category: parsed.category,
    priority: parsed.priority || 'Normal'
  },
  binary: binaryData // Attache le fichier PDF pour le prochain nœud HTTP POST
}];

3.3 Algorithme de Ranking (Scoring Backend)

Une fois les compétences extraites en JSON et reçues par le backend, le système doit calculer un score de matching. Ce processus a lieu dans le CandidateService.js.

  • Le système récupère le profil de poste (Job Posting) depuis la collection Jobs.
  • Il compare le tableau job.requiredSkills (ex: ["React", "Node.js", "MongoDB"]) avec le tableau candidate.extractedSkills fourni par l'IA.
  • Un algorithme calcule le pourcentage d'intersection. Les compétences obligatoires manquantes appliquent une pénalité sévère (-15%). Les années d'expérience (si supérieures au requis) ajoutent un bonus (+10%).
  • Le aiScore final (0 à 100) est sauvegardé dans MongoDB.

4.2 Implémentation du Tableau Kanban (react-beautiful-dnd)

Pour gérer visuellement le pipeline (Candidature reçue $\rightarrow$ En cours d'analyse $\rightarrow$ Entretien technique $\rightarrow$ Offre acceptée), nous avons implémenté un système de Drag-and-Drop complexe.

// rh-assistant-frontend/components/KanbanPipeline.tsx
import { DragDropContext, Droppable, Draggable } from 'react-beautiful-dnd';
import { useState } from 'react';
import api from '../lib/api';

export default function KanbanPipeline({ initialData }) {
    const [columns, setColumns] = useState(initialData);

    const onDragEnd = async (result) => {
        const { source, destination, draggableId } = result;

        // Si le glissement échoue en dehors des zones de dépôt
        if (!destination) return;
        
        // Si le candidat est déposé dans la même colonne à la même place
        if (source.droppableId === destination.droppableId && source.index === destination.index) return;

        // 1. Optimistic UI Update (Modification instantanée de l'état local)
        const newColumns = moveCardLocally(columns, source, destination);
        setColumns(newColumns);

        // 2. Synchronisation persistante avec le Backend
        try {
            await api.patch(`/api/v1/applications/${draggableId}/stage`, {
                newStage: destination.droppableId
            });
        } catch (error) {
            console.error("Échec du changement de statut, rollback...");
            // Si le serveur échoue (ex: panne internet), on annule le mouvement visuel
            setColumns(initialData); 
        }
    };

    return (
        <DragDropContext onDragEnd={onDragEnd}>
            {/* Rendu des colonnes Droppable et des cartes Draggable */}
        </DragDropContext>
    );
}

Lorsqu'un recruteur clique sur une carte Kanban, une page s'ouvre. Plutôt que d'obliger le recruteur à télécharger le PDF (comme le font les systèmes legacy), l'interface utilise un affichage scindé (Split-Pane via CSS Flexbox/Grid) :

5. DevOps, Déploiement et Stratégie de Tests Globale

5.1 Tests Exhaustifs

  • Tests Unitaires (Jest) : Application stricte sur les services mathématiques. Exemple : test('calculates correct score for exact skill match') vérifie que le service backend retourne bien 100%.
  • Tests d'API (Supertest) : Vérification que le routeur Express protège correctement les endpoints. Une fausse requête POST vers /api/jobs sans JWT doit obligatoirement renvoyer HTTP 401.
  • Stress Test de l'IA : Injection de 10 emails simultanés avec des PDF contenant des images complexes ou des polices inhabituelles pour observer le comportement de l'OCR de n8n et la gestion des Timeouts par l'API Groq.

5.2 Conteneurisation (Docker) et Architecture de Déploiement

Pour assurer un déploiement agnostique ("Ça marche sur mon ordinateur" $\rightarrow$ "Ça marche sur le serveur"), le backend et le frontend ont été conteneurisés en utilisant des fichiers Dockerfile multi-étapes (Multi-stage builds) afin de réduire le poids des images de production (en excluant devDependencies).

L'orchestration locale s'est faite via Docker Compose. En production, un serveur Linux Ubuntu est provisionné. Nginx agit comme un reverse proxy (passerelle inversée), capturant les requêtes HTTP (Port 80) et HTTPS (Port 443), gérant les certificats SSL via Let's Encrypt, puis dirigeant le trafic vers le conteneur Next.js (Port 3000) ou le conteneur API Node.js (Port 5000).

L'interface Next.js/Tailwind offre une expérience utilisateur de niveau entreprise, transformant les données algorithmiques abstraites en tableaux de bord Kanban pragmatiques et réactifs.
